\chapter*{Considerações Finais}\label{CapConclusoes}
\addcontentsline{toc}{chapter}{Considerações Finais}

Mediante os resultados obtidos e exibidos no Capítulo \ref{result}, fica evidente que é baixa a adoção do estilo de código descrito na Seção \ref{pep8}, se entendido como adoção o cumprimento à risca de todas as regras impostas pelo guia. Porém, se analisarmos pela quantidade de ocorrências, pode se entender que o resultado é aceitável se tratando de projetos onde o desenvolvedor não tem nenhum compromisso em estar aderente com o estilo de codificação proposto neste estudo. Se a comunidade Python desejar elevar o nível de adoção, é aconselhável que seja executada algumas ações que incentivem o desenvolvedor ou responsável pelo projeto a se adequar às regras. Uma sugestão para o incentivo pode ser através da submissão dos códigos-fonte a uma análise prévia antes de submetê-lo ao repositório. Esta análise, que pode ser semi ou completamente automática, poderia indicar a infração e apresentar sugestões de correção para o tipo de erro. 

Lembrando que este processo de análise não pode ser obrigatório, dadas as características e liberdades intrínsecas às comunidades de software livre., mas para motivar os usuários a  adequarem os seus projetos, poderia ser implementada uma classificação de usuários, onde quanto menos erros forem identificados no projeto a ser salvo no repositório melhor a classificação, gerando uma pontuação para cada projeto e uma pontuação total para o usuário. Estes usuários poderiam ser rankeados ou premiados de formas indiretas ou diretas.

Importante notar que mesmo tendo sido proposto pelo próprio criador da linguagem Python há quase 14 anos atrás, o PEP8 tem uma adoção modesta, conforme investigado no trabalho. Não é possível determinar, a partir do experimento realizado, se as violações do padrão se dão por desconhecimento do padrão ou por decisão consciente do desenvolvedor. Também vale notar que o PyPI não tem nenhuma regra ou orientação quanto à adoção de um estilo de codificação em particular, e que o nível de proficiência dos desenvolvedores que submetem códigos a esta base são os mais diversos.

Uma maior adoção de um estilo, mesmo que não o PEP8, certamente traria benefícios à comunidade como um todo, principalmente no que diz respeito à legibilidade e a manutenibilidade dos códigos. Contudo, sem ações que divulgem, orientem e incentivem tal adoção de um estilo uniforme, não será possível uma aderência ampla que resulte em tais benefícios.
